## 2024-05-09 - [CRITICAL] Prevent Command Injection in Process Executions
**Vulnerability:** Found insecure usage of `Process.Start(string)` and `ProcessStartInfo` where processes were launched without explicit restrictions, notably leaving `UseShellExecute` as default or `true`, which creates severe command injection and execution control risks.
**Learning:** `Process.Start` without enforcing `UseShellExecute = false` allows commands to be interpreted by the operating system shell. In contexts where process names or paths can be manipulated or aren't strictly hardcoded, this poses a risk of executing arbitrary commands.
**Prevention:** Always use `ProcessStartInfo` to launch processes and explicitly set `UseShellExecute = false` and `CreateNoWindow = true` (unless shell execution or window visibility are absolutely necessary and input is strictly sanitized).
## 2024-05-10 - [CRITICAL] Prevent Path Hijacking in System Utility Execution
**Vulnerability:** Found insecure usage of system utilities like `sc.exe`, `wmic.exe`, `powercfg.exe`, and `explorer.exe` being invoked via relative paths in `ProcessStartInfo.FileName`.
**Learning:** Using relative paths to execute commands leaves the application vulnerable to PATH hijacking (also known as a generic Local Privilege Escalation or LPE). An attacker could place a malicious executable with the same name earlier in the system's PATH environment variable, which would be executed instead of the intended system utility, often inheriting elevated privileges.
**Prevention:** Always use absolute paths (e.g., via `Environment.GetFolderPath(Environment.SpecialFolder.System)` and `System.IO.Path.Combine`) when starting any system utilities or external executables with `Process.Start`.
