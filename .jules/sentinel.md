## 2024-05-09 - [CRITICAL] Prevent Command Injection in Process Executions
**Vulnerability:** Found insecure usage of `Process.Start(string)` and `ProcessStartInfo` where processes were launched without explicit restrictions, notably leaving `UseShellExecute` as default or `true`, which creates severe command injection and execution control risks.
**Learning:** `Process.Start` without enforcing `UseShellExecute = false` allows commands to be interpreted by the operating system shell. In contexts where process names or paths can be manipulated or aren't strictly hardcoded, this poses a risk of executing arbitrary commands.
**Prevention:** Always use `ProcessStartInfo` to launch processes and explicitly set `UseShellExecute = false` and `CreateNoWindow = true` (unless shell execution or window visibility are absolutely necessary and input is strictly sanitized).

## 2024-10-26 - [CRITICAL] Prevent Path Hijacking in System Utility Executions
**Vulnerability:** Found insecure usage of system utility execution (like `wmic`, `explorer.exe`, `sc.exe`, `powercfg`) using relative paths or relying on system PATH resolution inside `ProcessStartInfo.FileName`.
**Learning:** Executing system utilities by simply relying on their filename instead of an absolute path makes the application vulnerable to Path Hijacking (a form of Local Privilege Escalation). An attacker can place a malicious executable with the same name in a directory that has a higher precedence in the system's PATH, or in the application's working directory, causing the malicious program to be executed instead of the intended system utility.
**Prevention:** Always use absolute, fully-qualified paths constructed using `Environment.GetFolderPath(Environment.SpecialFolder.System)` (or `Environment.SpecialFolder.Windows` where appropriate) combined with `Path.Combine()` when executing Windows system utilities via `Process.Start`.
